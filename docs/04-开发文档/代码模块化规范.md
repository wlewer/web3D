## 二、组件模块化原则

### 2.1 组件划分规则

```
┌─────────────────────────────────────────────────────────────────┐
│                     组件模块化划分                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 单一职责原则                                                │
│     每个组件只负责一个功能领域                                    │
│                                                                  │
│  2. 高内聚低耦合                                                │
│     组件内部紧密相关，组件之间依赖最小化                          │
│                                                                  │
│  3. 可复用性                                                    │
│     通用组件可被多个页面使用                                      │
│                                                                  │
│  4. 可测试性                                                    │
│     每个组件可独立测试                                           │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 组件分类

| 类型 | 目录 | 说明 | 示例 |
|:---|:---|:---|:---|
| 通用组件 | `components/common/` | 可复用的UI组件 | Button, Input, Modal |
| 布局组件 | `components/layout/` | 页面布局组件 | Header, Footer, Sidebar |
| 3D组件 | `components/3d/` | 3D渲染组件 | Scene, ModelViewer, Controls |
| 业务组件 | `components/business/` | 业务逻辑组件 | UserCard, WorkList |

### 5.2 服务编写规范

```typescript
// src/services/模块名.ts

import { api } from './api';

// 类型定义
interface RequestType {
  field1: string;
  field2: number;
}

interface ResponseType {
  id: string;
  name: string;
  data: any;
}

// API方法
export const serviceMethod = async (params: RequestType): Promise<ResponseType> => {
  try {
    const response = await api.post<ResponseType>('/endpoint', params);
    return response.data;
  } catch (error) {
    console.error('Service error:', error);
    throw error;
  }
};

// 导出所有方法
export const service = {
  method1: serviceMethod,
  // ...
};

## 六、禁止事项

### 6.1 禁止随意创建文件

```
❌ 禁止行为：

1. 在根目录创建.ts/.tsx文件
2. 在任意位置创建utils.ts等通用命名
3. 在components/直接创建组件（不分类）
4. 创建无人使用的组件
5. 创建重复功能的组件
```

### 6.2 禁止冗余结构

```
❌ 错误示例：

src/
├── components/
│   ├── Button.tsx              ❌ 应放在 common/
│   ├── MyButton.tsx            ❌ 应复用 Button
│   └── TestButton.tsx          ❌ 无人使用
│
├── utils/
│   ├── a.ts                    ❌ 命名不规范
│   ├── b.ts                    ❌ 命名不规范
│   └── helper.ts               ❌ 过于通用
```

### 6.3 正确的做法

```
✅ 正确示例：

src/
├── components/
│   ├── common/
│   │   └── Button.tsx          ✅ 规范命名
│   ├── layout/
│   │   └── Header.tsx
│   └── 3d/
│       └── Scene.tsx
│
├── utils/
│   ├── format.ts               ✅ 格式化工具
│   ├── validation.ts           ✅ 验证工具
│   └── three-helpers.ts        ✅ 3D辅助工具

## 七、检查清单

### 7.1 创建组件前

```markdown
□ 1. 是否已有相同功能的组件？
□ 2. 应该放在哪个分类目录？
□ 3. 是否需要创建新的分类？
□ 4. 组件是否可复用？
□ 5. 组件是否需要状态管理？
□ 6. 组件是否需要单元测试？
```

### 7.2 创建页面前

```markdown
□ 1. 是否需要创建页面专属组件目录？
□ 2. 是否需要创建页面专属hooks？
□ 3. 页面组件是否复用通用组件？
□ 4. API调用是否放在services目录？
□ 5. 类型定义是否统一管理？
```

### 7.3 代码审查

```markdown
□ 1. 目录结构是否符合规范？
□ 2. 文件命名是否规范？
□ 3. 组件是否单一职责？
□ 4. 是否有冗余代码？
□ 5. 是否复用现有组件？
□ 6. 是否有性能问题？
```

## 八、模块化示例

### 8.1 3D模块完整示例

```typescript
// src/components/3d/viewers/ModelViewer.tsx

import { Suspense } from 'react';
import { Canvas } from '@react-three/fiber';
import { OrbitControls, Environment, useGLTF } from '@react-three/drei';
import type { ModelFormat } from '../types';

interface ModelViewerProps {
  url: string;
  format: ModelFormat;
  autoRotate?: boolean;
  className?: string;
}

export const ModelViewer = ({ 
  url, 
  format, 
  autoRotate = false,
  className 
}: ModelViewerProps) => {
  return (
    <div className={className}>
      <Canvas>
        <Suspense fallback={null}>
          <Scene url={url} format={format} />
          <Environment preset="city" />
        </Suspense>
        <OrbitControls autoRotate={autoRotate} />
      </Canvas>
    </div>
  );
};
```
