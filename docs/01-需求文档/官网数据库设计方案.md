# 官网数据库设计方案

## 需求背景

官网目前使用localStorage存储相机配置，存在以下问题：
- 容量限制（约5MB）
- 无法跨设备同步
- 用户清除浏览器数据会丢失
- 模型列表硬编码在代码中

需要设计一个轻量级数据库方案，实现数据持久化。

---

## 方案对比

### 方案1：IndexedDB（推荐）

**特点**：
- 纯前端实现，无需后端
- 浏览器内置，零配置
- 容量大（50MB-2GB）
- 支持复杂查询和索引

**适用场景**：官网展示、个人使用

### 方案2：SQLite + 后端

**特点**：
- 跨设备同步
- 数据集中管理
- 需要后端服务器

**适用场景**：多用户、需要云端同步

### 方案3：JSON文件

**特点**：
- 最简单
- 直接读写文件
- 不适合大量数据

**适用场景**：极简需求

---

## 推荐方案：IndexedDB

### 优势

1. **零配置**：无需安装数据库服务器
2. **高性能**：异步API，不阻塞UI
3. **容量大**：50MB-2GB
4. **适合官网**：官网主要是展示性质

## 数据库表设计

### 1. 相机配置表（cameraConfigs）

| 字段 | 类型 | 说明 |
|------|------|------|
| modelId | string | 模型ID（主键） |
| position | [x, y, z] | 相机位置 |
| target | [x, y, z] | 观察点位置 |
| zoom | number | 缩放级别 |
| updatedAt | number | 更新时间戳 |

### 2. 模型信息表（models）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 模型ID（主键） |
| name | string | 模型名称 |
| modelUrl | string | GLB模型URL |
| splatUrl | string | SPZ模型URL |
| description | string | 描述 |
| category | string | 分类 |
| tags | string[] | 标签 |
| isFeatured | boolean | 是否精选 |
| sortOrder | number | 排序顺序 |
| thumbnail | string | 缩略图URL |
| updatedAt | number | 更新时间戳 |

### 3. 用户设置表（userSettings）

| 字段 | 类型 | 说明 |
|------|------|------|
| key | string | 设置键（主键） |
| value | any | 设置值 |
| updatedAt | number | 更新时间戳 |

## 技术选型依据

1. **轻量级**：idb库仅2KB，无依赖
2. **易用性**：Promise API，支持async/await
3. **兼容性**：所有现代浏览器支持
4. **性能**：异步操作，不阻塞UI
5. **容量**：50MB-2GB，满足需求
