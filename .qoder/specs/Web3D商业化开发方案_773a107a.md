## 一、现有能力 vs 商业需求差距分析

### 1.1 已满足的能力（可直接复用）

| 商业需求 | 现有实现 | 复用评估 |
|---------|---------|---------|
| 3D模型展示 | Base3DViewer + V3卡片组件，支持SPZ/GLB/PLY/SPLAT | 直接复用，需封装为插件 |
| 模板渲染引擎 | TemplateRenderer + ComponentRegistry + 懒加载 | 核心可复用，需升级为可视化搭建 |
| 用户认证 | JWT双Token + RBAC角色体系(4级) | 可复用，需增加多租户维度 |
| AI 3D生成 | 腾讯混元3D云端API(生产可用) + TripoSR | 直接复用 |
| 模型管理 | CRUD + 审核流 + 分类 + 标签 | 可复用，需增加租户隔离 |
| 国际化 | 中英文i18n（100+键） | 可复用 |
| 主题系统 | CSS变量注入 + Ant Design Token | 基础可复用，需扩展为租户级主题 |
| 容器化 | Docker + docker-compose(4服务) | 可复用，需增加K8s编排 |

### 1.3 技术债（需优先清理）

| 问题 | 影响 | 解决方案 |
|-----|------|---------|
| Python 3.12不兼容torchmcubes | TripoSR本地AI推理受限 | 使用云端API替代 |
| template_slots.is_dynamic NULL | Pydantic验证失败 | 数据库默认值修复 |
| 内存任务存储 | 重启丢失任务状态 | 迁移至Redis |
| JWT密钥硬编码 | 安全风险 | Vault/环境变量管理 |
| 单Zustand store | 复杂场景不足 | 按领域拆分store |

## 六、技术选型建议

| 领域 | 推荐方案 | 替代方案 | 理由 |
|-----|---------|---------|------|
| 可视化搭建 | react-grid-layout + react-dnd | GrapesJS | 与现有React生态融合 |
| 插件沙箱 | iframe + postMessage | Web Components | 安全隔离最彻底 |
| 支付 | 支付宝SDK + 微信支付SDK | Stripe | 国内市场优先 |
| 访客分析 | Umami(自托管) | Plausible | 开源免费+数据自主 |
| CI/CD | GitHub Actions | GitLab CI | 与GitHub仓库集成 |
| K8s | 阿里ACK/腾讯TKE | 自建K8s | 降低运维复杂度 |
| CDN | 又拍云/腾讯CDN | CloudFront | 国内加速 |
| 监控 | Prometheus+Grafana | Datadog | 开源+自定义灵活 |
| SEO | Next.js SSR 或 Prerender | Gatsby | 动态内容需SSR |
